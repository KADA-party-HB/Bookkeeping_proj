NEXT.JS + ORM REBUILD BRIEF
===========================

Purpose
-------
This project is a rental management system for tents and event equipment. It supports public browsing, customer registration/login, quote-based booking creation, admin inventory management, pricing management, and booking administration.

The important part is not the original file tree. The important part is the system structure, domain model, and business rules below. Another AI should be able to rebuild the project from this document alone.


1. SYSTEM OVERVIEW
------------------
The application manages real physical rental inventory.

It is not a simple "stock quantity" system.

Each rentable item is an actual physical unit with its own identity. Bookings allocate specific items for a date range. Availability is therefore calculated by checking whether those exact items are already used in overlapping bookings.

The app has four main areas:

1. Public site
- Home page
- About page
- FAQ page
- SEO assets like sitemap, robots, favicon, manifest

2. Authentication
- Register customer account
- Login/logout
- Session-based role handling

3. Customer-facing booking flow
- Choose rental dates
- See available categories for those dates
- Select quantities by category
- Optionally add delivery
- Optionally add setup service
- Create booking
- View booking details and booking history

4. Admin back office
- Manage customers
- Manage bookings
- Manage categories
- Manage physical items
- Manage rental periods
- Import/export pricing
- Reallocate booking items when dates or item selections change
- Review booking metrics and booking calendar


2. USER ROLES
-------------
There are two roles:

1. Customer
- Can register and log in
- Has a linked customer profile
- Can create bookings
- Can view their own bookings

2. Admin
- Can access management pages
- Can create and edit customers
- Can create and edit bookings
- Can create and edit items and categories
- Can manage rental periods and pricing
- Can confirm, cancel, and delete bookings

4. CRITICAL BUSINESS RULES
--------------------------
These rules are the heart of the project and must survive the rewrite.

1. Physical inventory rule
- Inventory is tracked per item, not per category quantity.
- A booking reserves actual items.

2. Shared stock rule
- One item can belong to multiple categories.
- Those categories share the same underlying stock because they may point to the same physical item.

3. Booking item validity rule
- A booking item stores both item_id and category_id.
- The pair (item_id, category_id) must exist in the membership table.

4. Overlap prevention rule
- The same physical item cannot be allocated to two overlapping active bookings.
- Cancelled bookings do not block availability.

5. Rental pricing rule
- Price is determined by category plus rental period.
- Rental period is matched from booking duration in days.

6. Category subtype rule
- A category must behave as either tent or furnishing.
- Tent categories have tent-specific fields including setup fee.
- Furnishing categories have furnishing-specific fields.

7. Price snapshot rule
- Booking items store quoted rental period label and quoted price at booking time.
- This prevents old bookings from changing when category pricing changes later.

8. Override rule
- Admin can override line totals and booking total.
- Override notes must be stored.

9. Delivery rule
- Delivery is optional.
- Delivery fee is derived from route distance, with a base fee, included distance, and per-km extra fee.
- The resolved delivery address and distance are stored on the booking.

10. Setup service rule
- Setup service is optional at booking level.
- Tent category setup fee contributes to totals when enabled.

11. Pending hold rule
- Pending bookings reserve inventory temporarily.
- Stale pending bookings are automatically expired after a configurable number of days.
- A customer can only have a limited number of active pending bookings.


5. MAIN WORKFLOWS
-----------------
Recreate these workflows in the new app.

1. Registration
- Customer signs up with name, email, phone, and password.
- A user account and customer profile are created together.
- Registration is blocked if the email already belongs to an existing customer profile that was pre-created by admin.

2. Login/logout
- Email/password authentication
- Session-based auth in the current app
- In Next.js, use a clean auth layer but preserve roles and customer/admin separation

3. Public availability search
- User enters start date and end date
- System calculates rental days
- System returns categories that are available for that date range
- Each category response should include:
  category info
  subtype info
  matched rental period pricing
  count of available items
  item ids that are available
  item ids that are same-day turnaround only

4. Guest booking creation
- Guest selects date range and category quantities
- Optional delivery quote is requested from the delivery service
- Booking is created with allocated items
- Inventory is reserved immediately as a pending booking

5. Customer booking creation
- Logged-in customer can create bookings from the site
- Same allocation and pricing rules as guest flow

6. Admin booking creation/editing
- Admin can create bookings on behalf of customers
- Admin can manually pick or reassign allocated items
- Admin can edit dates, delivery, setup, notes, and price overrides
- If dates change, allocation must be revalidated/recomputed

7. Customer management
- Admin can list customers with sorting and pagination
- Admin can create customers manually
- Admin can edit customer details
- Customer detail page shows booking history

8. Item management
- Admin can create item with SKU and active flag
- Admin can assign one or more categories to an item
- Item cannot be deleted if it is tied to active or future bookings

9. Category management
- Admin can create tent category
- Admin can create furnishing category
- Admin can edit category base fields and subtype fields
- Admin can manage category pricing across rental periods

10. Rental period management
- Admin can create/edit/delete rental periods
- Deletion should be blocked or guarded if the rental period is in use

11. Booking lifecycle
- Pending booking can be confirmed
- Booking can be cancelled
- Cancelled booking can be deleted
- Booking detail page shows line items, pricing breakdown, and notes

12. Pricing sync
- Admin can export category pricing as JSON
- Admin can import JSON and apply pricing updates back into the database

7. HOW TO STRUCTURE THE NEXT.JS + ORM VERSION
---------------------------------------------
Use Next.js App Router with PostgreSQL and Prisma ORM.

Suggested structure:

- `app/`
  pages, route groups, layouts, server actions, and route handlers

- `components/`
  reusable UI components for forms, tables, booking summaries, category cards, etc.

- `lib/db.ts`
  Prisma client singleton

- `lib/auth/`
  auth/session helpers, role guards, current-user helpers

- `lib/repositories/`
  low-level ORM queries grouped by domain:
  users
  customers
  categories
  items
  rental periods
  bookings

- `lib/services/`
  business logic that coordinates repository calls:
  booking service
  allocation service
  pricing service
  delivery service
  mail service
  customer service

- `lib/validators/`
  zod schemas or equivalent for forms and actions

- `prisma/schema.prisma`
  full relational schema

- `prisma/seed.ts`
  demo/admin/customer seed data

- `prisma/migrations/`
  database history

- `public/`
  images, PDFs, and static assets


8. ORM DESIGN NOTES
-------------------
Prisma should handle most of the schema and CRUD.

However, do not force every part into simple ORM queries.

Use Prisma transactions and, if needed, carefully targeted raw SQL for:
- overlap-safe item allocation
- lock-based candidate selection
- complex availability queries across bookings and items
- any query where correctness under concurrency matters more than purity

This project has real inventory allocation logic. That is the place where naive ORM-only code often becomes incorrect.
